# План: Завтракаст СДВГ

## Подход

Собрать русскоязычное одностраничное React/TypeScript-приложение, в котором вся продуктовая логика и подготовленные данные статичны, а единственный сетевой runtime-источник — официальные удалённые MP3. Метаданные брать build-time из публичного WordPress API Завтракаста, потому что RSS содержит только последние 100 выпусков; транскрипцию выполнять возобновляемым локальным пайплайном через Whisper API, а темы и исключения хранить в проверяемом редакционном JSON.

Интерфейс строится вокруг одного плеера и двух независимых локально сохраняемых очередей. Для PWA оболочка, локальные обложки, каталог и все транскрипты кэшируются после первого открытия, а аудио всегда остаётся `NetworkOnly`.

## Зафиксированная стартовая выборка

Алгоритм `one-per-year-v1` группирует подходящие записи WordPress API категории `2` по локальному году, стабильно сортирует их по `(date, link)` и выбирает индекс `uint32(first8hex(SHA256("zavtracast-sdvg:v1:" + year))) % count`.

## Этапы

### 1. Зафиксировать цель и создать статический каркас

Файлы и системы:

- сохранить `goals/zavtracast-sdvg/{facts.md,facts.meta.json,facts-result.json,plan.md,goal.md}`;
- создать React/TypeScript-каркас и hosting-конфигурацию в корне репозитория;
- удалить демонстрационный, auth- и database-код стартера, оставив одну клиентскую страницу и hosting-адаптер;
- настроить `package.json`, TypeScript, ESLint, Vitest и Playwright;
- добавить `.gitignore` для `.cache/audio`, аудиочанков, временных Whisper-ответов и секретов.

Проверка:

- `npm ci`;
- `npm run typecheck`;
- `npm run build`;
- исходники runtime не содержат API-маршрутов, базы данных или аутентификации.

Коммит: `chore: scaffold static Zavtracast SDVG app`.

### 2. Реализовать схему данных и воспроизводимый выбор выпусков

Файлы и системы:

- `src/domain/schema.ts` — схемы каталога, метаданных, фраз, тем и исключений;
- `scripts/archive.ts` — загрузка всех страниц `wp-json/wp/v2/posts?categories=2`, извлечение `contentUrl`, обложки, длительности и шоунотов;
- `scripts/select-episodes.ts` — фильтрация выпусков, стабильная выборка `one-per-year-v1`;
- `data/selection.json` — seed, снимок источника, алгоритм и 12 выбранных выпусков;
- `data/episodes/<id>/metadata.json`;
- `tests/data/selection.test.ts`, `tests/data/schema.test.ts`.

Проверка:

- повторный `npm run data:select -- --seed zavtracast-sdvg:v1` даёт ту же выборку;
- валидатор доказывает ровно один выпуск для каждого года 2015–2026;
- у всех 12 записей есть корректные страницы, обложки, длительности и точные MP3 URL;
- диапазонный HTTP-запрос к каждому MP3 возвращает аудиоданные и поддержку перемотки.

Коммит: `feat: add reproducible episode selection`.

### 3. Реализовать возобновляемый Whisper-пайплайн

Файлы и системы:

- `scripts/fetch-assets.ts` — загрузка MP3 в игнорируемый `.cache/audio` и обложек в `public/covers`;
- `scripts/split-audio.ts` — `ffmpeg silencedetect`, разбиение около 10–15 минут по ближайшей тишине и mono 16 kHz WAV;
- `scripts/transcribe.ts` — последовательные multipart-запросы к `/v1/audio/transcriptions` с `language=ru`, `response_format=verbose_json`, настраиваемыми `WHISPER_API_BASE` и `WHISPER_MODEL`;
- `.env.example` с рекомендуемыми `http://192.168.3.21:12017` и `large-v3-turbo-q5_0`;
- отдельный атомарный результат каждого чанка, checksum и resume без повторной обработки;
- `scripts/normalize-transcript.ts` — прибавление абсолютных смещений, проверка стыков и объединение Whisper-сегментов до границ конечной пунктуации;
- `data/episodes/<id>/transcript.json`;
- `docs/data-pipeline.md` и mocked integration-тест контракта Whisper.

Проверка:

- тестовый чанк возвращает `verbose_json` с монотонными `start/end`;
- прерванный прогон продолжается с первого отсутствующего чанка;
- нормализация не создаёт произвольный таймкод внутри Whisper-сегмента, не теряет текст на стыках и отклоняет пустые/немонотонные ответы;
- для каждого полного выпуска последний таймкод попадает в его длительность.

Коммит: `feat: add resumable Whisper transcription pipeline`.

### 4. Подготовить редакционную разметку и playback-каталог

Файлы и системы:

- `data/episodes/<id>/editorial.json` — созданные Codex темы, краткие описания и исключённые intro/music/ad/donations/thanks/technical-pause диапазоны;
- `scripts/validate-data.ts` — существование ссылочных phrase ID, точное совпадение времён с границами фраз, порядок, диапазоны, отсутствие пересечений тем с исключениями;
- `src/domain/minute-clips.ts` — детерминированные неперекрывающиеся 45–75-секундные фрагменты, ближайшие к 60 секундам и ограниченные завершёнными фразами;
- `scripts/build-playback-index.ts`;
- `public/data/catalog.json`, `public/data/episodes/<id>.json`;
- `tests/data/editorial.test.ts`, `tests/unit/minute-clips.test.ts`.

Проверка:

- `npm run data:validate` проходит на всех 12 выпусках;
- каждая тема и минута находится внутри длительности выпуска и вне исключений;
- каждый минутный элемент длится 45–75 секунд и начинается/заканчивается на сохранённой границе фразы;
- сгенерированный каталог имеет content fingerprint и ссылается на все 12 транскриптов.

Коммит: `feat: add twelve curated episode transcripts`.

### 5. Реализовать очереди и контроллер плеера

Файлы и системы:

- `src/domain/queue.ts` — независимое состояние `topics`/`minute`, Fisher–Yates, отсутствие повторов в цикле, защита от повтора на стыке циклов, назад/вперёд и миграция по dataset fingerprint;
- `src/domain/playback.ts` — `playIntent`, загрузка → metadata → seek → play, точное завершение диапазона, один автоматический переход, pause/retry/skip, volume и clamped seek;
- `src/lib/persistence.ts` — версионированное локальное состояние;
- `src/features/player/*`;
- fake-audio unit/component tests.

Проверка:

- тесты доказывают уникальность полного цикла, сохранение обеих очередей и истории после перезагрузки;
- без первого пользовательского действия аудио не запускается;
- пауза отменяет автопродолжение, конец диапазона переводит ровно на один следующий элемент;
- media error не сдвигает очередь до явного «Следующий».

Коммит: `feat: build random segment playback engine`.

## Риски и решения

- **Авторские права.** Аудио не копируется в приложение и проигрывается с официальных URL, но публикация полных производных транскриптов может требовать разрешения правообладателя. До публичного распространения нужно подтвердить разрешение; технически публикацию можно оставить приватной.
- **Антибот и имена MP3.** Имена файлов нельзя выводить из номера выпуска. Пайплайн всегда извлекает `itemprop="contentUrl"` из WordPress content и проверяет range-запрос; уже подтверждено, что точные URL поддерживают byte ranges.
- **Длинная транскрибация.** Сервер принимает тело до 1 ГБ, но декодирует весь запрос в PCM и выполняет запросы последовательно. Поэтому используются короткие чанки, тишина на границах, атомарные результаты и resume.
- **Нет словных таймкодов.** `timestamp_granularities[]=word` сервер игнорирует. Граница фразы разрешена только между Whisper-сегментами; смежные сегменты объединяются до конечной пунктуации, а сомнительные стыки блокируют валидацию.
- **Изменение архива.** Выборка фиксируется seed, source ID и URL в Git; изменения внешнего API не меняют уже выпущенный каталог молча.
- **Офлайн-объём.** Все 12 транскриптов измеряются на build; файл выше установленного лимита завершает сборку ошибкой, а не исчезает из precache.
- **Развёртывание и статичность.** Hosting-адаптер может использовать платформенный worker для выдачи ассетов, но приложение не имеет runtime API, базы или серверной бизнес-логики; это проверяется сканированием production bundle и e2e.
